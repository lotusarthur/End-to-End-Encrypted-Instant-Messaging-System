3. 消息传输格式（完整版）

----------------------

传递格式:

- 基础消息格式 (Message对象，请不要使用它！这是过时的！):

  Message(message_id, from_user, to_user, ciphertext, timestamp, ttl_seconds, status, signature)



- 加密网络包格式 (EncryptedNetworkPackage):

  EncryptedNetworkPackage(

      message_id, sender_id, receiver_id, ciphertext_b64, 

      nonce_b64, mac_tag_b64, ad_serialized, timestamp, ttl_seconds, status

  )



- WebSocket实时消息格式:

  {

      "type": "message",

      "message_id": "...",

      "sender_id": "...", 

      "receiver_id": "...",

      "ciphertext_b64": "...",

      "nonce_b64": "...",

      "mac_tag_b64": "...",

      "ad_serialized": "...",

      "timestamp": ...,

      "ttl_seconds": ...,

      "status": "delivered"

  }



实现思路:

- 支持新旧两种消息格式的兼容

- 确保加密字段的Base64编码正确性

- 实时消息和离线消息的统一处理



4. 好友关系管理格式（完整版）

--------------------------

传递格式:

- 好友申请发送: db_manager.add_friend_request(from_user, to_user) -> request_id

- 获取好友申请: 

  db_manager.get_friend_requests(username, "received")  # 收到的申请

  db_manager.get_friend_requests(username, "sent")     # 发送的申请

  

- 申请状态更新: db_manager.update_friend_request_status(request_id, status)

  状态值: "pending", "accepted", "declined", "cancelled"

  

- 好友列表获取: db_manager.get_friends(username)

- 好友关系删除: db_manager.remove_friend(user_a, user_b)



实现思路:

- 确保好友关系的双向一致性

- 处理重复申请和状态冲突

- 验证时间戳和状态转换的正确性



5. 非好友通信限制格式

-------------------

传递格式:

- 消息发送前验证: message_manager.can_send_message(sender, receiver) -> bool

- 好友关系检查: 通过FriendRelationship表验证

- 错误响应格式: {"error": "not_friends", "message": "需要先成为好友才能发送消息"}



实现思路:

- 在消息发送前进行好友关系验证

- 对非好友消息进行拦截和错误提示

- 支持好友申请流程的完整性



6. 消息状态管理格式

-----------------

传递格式:

- 消息发送: message_manager.mark_message_sent(message_id, sender, receiver)

- 消息投递: message_manager.mark_message_delivered(message_id)

- 状态查询: db_manager.get_offline_messages(username)

- 状态更新: db_manager.update_message_status(message_id, new_status)



实现思路:

- 维护消息的完整生命周期状态

- 支持离线消息的缓存和拉取

- 确保状态转换的一致性
